# US-018.5 — Regra de conflito (Planilha vs Supabase)

Data: 2026-04-30

## Objetivo

Garantir que a sincronização (Planilha → Supabase) **não sobrescreva decisões operacionais** tomadas no Supabase durante a transição.

Resumo da regra recomendada:

- **Dados cadastrais**: podem ser atualizados pela planilha durante a transição (com regras de merge).
- **Dados operacionais**: **somente Supabase** pode alterar. Importação da planilha nunca sobrescreve.

## Conceitos

### Origem do registro

Campos (US-018.4) usados para auditoria e controle:

- `origem_dado`: `PLANILHA` | `SISTEMA`
- `data_importacao`
- `id_origem_planilha`
- `ultima_sincronizacao`
- `criado_via_sistema`

### “Dado cadastral” vs “dado operacional”

**Cadastral (Planilha pode atualizar):**

- `nome`
- `telefone` / `whatsapp`
- `email`
- `bairro`
- `data_nascimento`
- `endereco`
- `responsavel_*` (nome/telefone/email)
- `observacoes_cadastrais` (se existir)

**Operacional (somente Supabase):**

- `status_inscricao` / `status`
- `prioridade`
- `fila` / `waitlist`
- `confirmado`
- `nao_selecionado`
- `presenca` / `checkin`
- `historico_status`
- `usuario_alteracao`
- `data_alteracao`
- `auditoria` (logs, timestamps internos)

## Regra de prioridade (campo a campo)

### 1) Importação de planilha (sync)

Quando o sync tentar atualizar um registro existente no Supabase:

1. **Sempre atualizar**:
   - `ultima_sincronizacao = now()`
   - `origem_dado = 'PLANILHA'` (se o registro for importado)
   - `data_importacao` (se for primeira importação)
   - `id_origem_planilha` (se disponível)
2. **Atualizar apenas campos CADASTRAIS**.
3. **Nunca atualizar campos OPERACIONAIS**.
4. Se o payload da planilha trouxer valor divergente em campo OPERACIONAL, isso é **conflito**:
   - manter o valor do Supabase
   - registrar log de conflito

### 2) Alteração feita pelo sistema (UI / API)

1. Registros criados pela UI/API devem ter:
   - `origem_dado = 'SISTEMA'`
   - `criado_via_sistema = true`
2. Alterações operacionais (fila, presença, confirmação, prioridade) são **sempre Supabase**.

### 3) Caso especial: registro `SISTEMA` que aparece depois na planilha

Para evitar sobrescrita indevida:

- A planilha pode **complementar** campos cadastrais **somente se estiverem vazios** no Supabase (merge “fill only”).
- Campos operacionais continuam protegidos.

## Log de divergência (aceite)

Sempre que houver divergência entre planilha e Supabase em campo **operacional**, registrar:

- entidade/tabela
- chave do registro
- campo
- valor_supabase
- valor_planilha
- timestamp
- id da execução do sync (se existir)

Implementação sugerida no banco em `docs/US-018.5-regra-conflito-sheets-vs-supabase.sql`.

## Whitelist por entidade (sugestão inicial)

Adapte os nomes exatamente aos da sua tabela base (não à view).

| Entidade (Supabase) | Campos “CADASTRAIS” (planilha pode atualizar) | Campos “OPERACIONAIS” (somente Supabase) |
|---|---|---|
| `cadastro_oficial` / `pessoas` / `adolescentes` | `nome`, `telefone`, `email`, `bairro`, `data_nascimento`, `endereco`, `responsavel_*` | `status_*`, `prioridade`, `fila`, `confirmado`, `historico_status`, `usuario_alteracao`, `data_alteracao` |
| `nao_inscritos` | `nome`, `telefone`, `email`, `bairro`, `data_nascimento`, `recado` (se for cadastral) | `status_inscricao`, `prioridade`, `fila`, `confirmado`, `nao_selecionado`, `status_pre_confirmacao` |
| `inscricoes_prioritarias` / `inscricoes` | `nome`, `telefone`, `email`, `bairro`, `data_nascimento` | `status_inscricao`, `prioridade`, `confirmado`, `nao_selecionado`, `presenca` |
| `encontreiros` | `nome_completo`, `email`, `celular_whatsapp`, `bairro`, `endereco_completo` | `elegivel`, `status_equipe`, `historico_status` |
| `eventos_agenda` | `atividade`, `tipo`, `inicio`, `termino`, `local` *(cuidado: se o sistema edita, tratar como operacional)* | `status`, `usuario_alteracao`, `data_alteracao` |
| `comunicados` | `titulo`, `assunto`, `corpo`, `data_agendada` | `status_operacional`, `usuario_alteracao`, `data_alteracao` |
| `presencas` | *(nenhum — presença é operacional)* | **tudo** (check-in/presença é Supabase) |
| `disparos/*` | *(nenhum — disparos são operacionais)* | **tudo** |

## Mapeamento para suas views atuais

Você informou que já existem views prontas. Recomendação:

- O **sync** deve escrever nas **tabelas base** (`cadastro_oficial`, `nao_inscritos`, `inscricoes_prioritarias`, etc.).
- O painel pode ler das **views** (ex.: `vw_cadastro_oficial`, `vw_nao_inscritos`, `vw_inscricoes_prioritarias`, `vw_distribuicao_circulos`, `vw_presencas_historico`), mas as regras de conflito precisam estar na origem (tabela + função/trigger).

## Onde isso aparece no sistema

- Logs podem ser consultados pela tela `logs` (se você publicar o log em `logs_auditoria` ou expor um endpoint/view).
- Origem do registro (PLANILHA/SISTEMA) fica visível nos modais/drawers via `components/DataOriginAudit.tsx`.
