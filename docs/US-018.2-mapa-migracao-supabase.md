# US-018.2 — Mapa de migração (Sheets -> Supabase)

Data: 2026-04-30

Este documento mapeia as telas atuais do painel EAC v2, indicando a fonte operacional de dados (Sheets vs Supabase), a fonte futura desejada e as tabelas/aliases esperadas no Supabase.

## Regra de validação (obrigatória)

Nenhuma tela deve ser considerada “migrada” (principalmente em **gravação**) sem validar a existência/estrutura da(s) tabela(s) correspondentes no Supabase.

No código atual, as leituras principais passam por `/api/comunicados` e, quando o Supabase está configurado, **não** caem para planilha por padrão.

## Ações já atendidas via Supabase (leitura)

Hoje o backend atende via Supabase as ações:

- `GET_SYNC_STATUS`
- `USER_LOGIN`
- `GET_MEMBERS`, `SEARCH_MEMBERS`
- `GET_NON_ENROLLED`
- `GET_EMAIL_STATUS_SUMMARY`
- `GET_INSCRICOES_PRIORITARIAS`
- `GET_CIRCULOS_DISTRIBUIDOS`
- `GET_PRESENCE`
- `GET_ENCONTREIROS`
- `GET_EVENTS`
- `GET_COMUNICADOS`
- `GET_LOGS`

| Presença (`presence`) | Leitura via Supabase (`GET_PRESENCE`) + **check-in em Sheets** (`MARK_PRESENCE`) | Supabase (leitura + check-in) | `EAC_SUPABASE_TABLE_PRESENCE` ou `controle_presenca/presenca/presence` | Parcial (leitura OK; gravação pendente) |
| Disparos (`dispatches`) | Execuções em **Sheets/Apps Script** (`EXECUTE_*`, `CLEAR_DISPATCH_STATUS`, etc.) | Supabase (jobs/filas/logs) ou serviço dedicado | (a definir) | Não migrado |
| Agenda/Calendário (`calendar`) | Leitura via Supabase (`GET_EVENTS`) + **CRUD em Sheets** (`SAVE_EVENT`, `DELETE_EVENT`) | Supabase (leitura + CRUD) | `EAC_SUPABASE_TABLE_EVENTS` ou `eventos/events/calendar_events` | Parcial (leitura OK; gravação pendente) |
| Comunicados (`comunicados`) | Leitura via Supabase (`GET_COMUNICADOS`) + **CRUD em Sheets** (`SAVE_COMUNICADO`, `DELETE_COMUNICADO`) | Supabase (leitura + CRUD) | `EAC_SUPABASE_TABLE_COMUNICADOS` ou `comunicados/announcements/notificacoes` | Parcial (leitura OK; gravação pendente) |

## Como validar tabela no Supabase (mínimo)

Antes de “virar” uma operação, valide:

1) a tabela existe (ou view) e tem as colunas mínimas que o frontend usa
2) a sincronização está populando dados recentes (timestamp/updated_at)
3) RLS/políticas permitem o acesso esperado (para server-side, preferir service role)

Sugestão de query (ajuste nomes):

```sql
select count(*) from cadastro_oficial;
select max(updated_at) from cadastro_oficial;
```
